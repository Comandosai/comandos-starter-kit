# Создание .env (для агента) — ОБЯЗАТЕЛЬНО, не только пример

Раньше частая ошибка: оставляли только `.env.example` → проект не запускался.
Теперь правило: агент СОЗДАЁТ рабочий `.env`.

## Имя env-файла
По умолчанию `.env`. Если в рецепте задано поле `env_file` (напр. `.env.local` для Next.js —
saas/miniapp) — используй его. `.gitignore` уже покрывает `.env` и `.env.*`.

## Алгоритм
1. Есть `.env.example` (template-ветки) → скопируй в env-файл. Нет (scaffold_cmd, напр.
   create-next-app не создаёт пример) → создай env-файл напрямую из списка `needs.env`.
2. Для каждого ключа из `needs.env` рецепта:
   - есть в окружении/у резидента — впиши значение;
   - нет — спроси у резидента человеческим языком, где взять подскажи (`lib/mcp-install.md`).
3. Если резидент пока не готов дать ключ — оставь строку пустой и явно скажи:
   «Впишешь ключ в `.env` позже, без него запуск не сработает».
4. Убедись, что `.env` покрыт `.gitignore` (он создаётся из `gitignore.tmpl`).
5. НИКОГДА не коммить `.env` и не печатать ключи в чат/логи.

## Проверка
После создания `.env` агент проверяет, что нужные переменные видны при запуске
(скрипты шаблонов используют `--env-file-if-exists=.env`).
