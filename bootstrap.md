## Принципы (запомни на всю сессию)

1. **Convention, not catalog.** Резидент может захотеть что угодно. Твоя задача —
   не «выбрать пункт меню», а понять намерение и собрать по стандартам COMANDOS.
2. **Не ломай существующее.** Если проект уже есть — изучи и уважай его структуру.
3. **Один язык — TypeScript** для всех продуктов (дефолт). Python — только если
   задача требует тяжёлого ML и резидент это подтвердил.
4. **Всегда оставляй след:** в созданном проекте должен быть `AGENTS.md` и
   `PROJECT-BRIEF.md`, чтобы следующая сессия знала контекст.
5. **Никогда не показывай резиденту литеральные плейсхолдеры** (`{{...}}`, `<TODO>`).
   Если значения нет — спроси по-человечески или подставь разумное.

## ШАГ 1. DOCTOR — выровнять окружение

Запусти доктор. Он проверяет **только**: CLI (node/git), какие IDE стоят,
кандидатов базы знаний, наличие config. **MCP и ENV доктор НЕ проверяет** —
это ты делаешь сам на ШАГЕ 5 по `lib/mcp-install.md`.

- **macOS / Linux:** `bash ~/.comandos-kit/doctor/doctor.sh`
- **Windows:** `powershell -ExecutionPolicy Bypass -File ~/.comandos-kit/doctor/doctor.ps1`
  (если `~` не разворачивается — подставь полный путь к домашней папке)
- **bash и powershell недоступны:** выполни сам, через диалог и доступные инструменты,
  ВСЕ 4 проверки: (1) есть ли node и git; (2) какие IDE; (3) поиск базы знаний;
  (4) есть ли `~/.comandos/config.json`.

### Если node или git = MISSING — НЕ продолжай молча
Объясни резиденту простыми словами и поставь (см. `lib/mcp-install.md` → «CLI»):
- на чистом Mac первым делом может всплыть окно «установить инструменты разработчика» —
  это нормально, попроси нажать «Установить» и подождать.
- доведи до того, что `node -v` и `git --version` отвечают, и только потом дальше.

### Проверь версию Node ≥ 22.7
Шаблоны используют флаги `--env-file-if-exists` и type-stripping (нужен Node ≥ 22.7).
Сравни версию из отчёта doctor: если ниже — попроси обновить Node до текущего LTS
(nodejs.org / `brew upgrade node`) простыми словами, прежде чем запускать проект.

## ШАГ 3. База знаний (vault) — найти и привязать

1. Возьми кандидатов из отчёта doctor (строки `candidate: ...`).
2. Нашёл → спроси: **«Это твоя база знаний? <путь> — да/нет»**.
   - Да → запиши путь в `knowledge_base` конфига. Проект будет туда ссылаться.
3. Не нашёл / ответ «нет» → предложи на выбор:
   - **(a, рекомендуется)** «У тебя настроен Второй мозг из курса COMANDOS?
     Если нет — попроси у куратора COMANDOS ссылку на модуль «Второй мозг» и настрой,
     тогда все проекты будут компаундить знания.»
   - **(b)** «Создать простую базовую папку для знаний прямо сейчас?» →
     создай `<projects_dir>/_knowledge/` с `README.md`.
   - **(c)** «Пропустить — проект будет standalone.» → `knowledge_base: null`.
4. Запиши выбор в конфиг. Больше не спрашивай.

## ШАГ 4. ИНТЕРВЬЮ — что делаем

Спроси дружелюбно (по одному вопросу, не вываливай всё сразу):

1. **«У тебя уже есть папка/проект, который правим, — или начинаем с нуля?
   Если не уверен, просто опиши задачу, я пойму сам.»**
   - Есть существующий → перейди к ШАГУ 4-Б.
2. **«Опиши своими словами, что хочешь сделать.»**
3. По ответу определи ВЕТКУ (см. `recipes/_schema.md`):
   - Разовый артефакт (презентация/видео/картинка/гайд) → ветка A
   - Продукт (сайт/бот/агент/SaaS/mini-app) → ветка B
   - Знание (research/конспект) → ветка C
4. Подбери рецепт из `recipes/` **детерминированно**:
   1) Точное совпадение рецепта → бери его.
   2) Точного нет, но есть строка в палитре (`recipes/_schema.md`) → собери по палитре.
   3) Нет ни рецепта, ни строки палитры → проверь, есть ли профильный навык в твоей
      среде (напр. browser-extension-builder); есть → используй его; нет → честно
      скажи резиденту и предложи ближайший разумный стек, подтверди с ним.
5. Задай уточняющие вопросы из поля `ask` рецепта.

### ШАГ 4-Б. Существующий проект (доработка)
1. **Узнай, ГДЕ проект.** Спроси: «Дай путь к папке проекта (или назови имя — поищу
   в `<projects_dir>`)». Не знает — просканируй `projects_dir`, покажи список, дай выбрать.
   Подтверди путь, прежде чем что-либо делать.
2. **Безопасность ПЕРЕД правками.** Проверь `git status` в папке проекта:
   - нет git → предложи сделать копию папки или `git init` перед изменениями;
   - есть незакоммиченные изменения → предупреди, предложи сначала зафиксировать.
3. **Картируй проект.** Прочитай все манифесты, какие есть (`package.json`,
   `pyproject.toml`/`requirements.txt`, `astro.config.*`, `next.config.*`, `.csproj` и т.п.),
   `README`, `AGENTS.md`/`CLAUDE.md`, верхние папки (`src/app/docs/...`), `.gitignore`.
   Определи стек и конвенции. Сохрани карту в `~/.comandos/maps/<имя-проекта>-map.md`
   (глобально, чтобы не засорять чужой проект; создай папку `~/.comandos/maps/`, если её нет).
4. **Уточни задачу доработки** (что добавить/починить).
5. **Внеси изменения по карте**, не меняя структуру.
6. **Покажи `git diff`** человеческим языком и как проверить результат.
7. Обнови `PROJECT-BRIEF.md`/`AGENTS.md`, если они есть; нет — предложи создать.
   → На этом для доработки ПЕРЕХОДИ к ШАГУ 6 (ветка «существующий проект»).

## ШАГ 5. РЕЦЕПТ — развернуть

Цель для нового проекта: `target = <projects_dir>/<slug-имени-проекта>`
(slug — латиницей, через дефис). Создай папку, разворачивай туда.

**Сначала сверь `needs` рецепта** (cli/mcp/env) с отчётом doctor и реальностью:
- CLI которого нет — доустанови (`lib/mcp-install.md`).
- MCP из `needs.mcp` — проверь, подключён ли (`claude mcp list` / конфиг IDE);
  нет → установи по `lib/mcp-install.md` (раздел нужной IDE), при необходимости спроси ключ.
- ENV из `needs.env` — чего нет, спроси у резидента по-человечески (где взять — подскажи),
  сохрани в `.env` создаваемого проекта (он в `.gitignore`).

**Разворачивание — по типу источника рецепта (ровно одно поле):**

- **`template`** (папка кода в kit) → скопируй `~/.comandos-kit/<template>/` в `target`.
  Если папки нет → НЕ падай: собери скелет по «Открытому пути» (`recipes/_schema.md`).
- **`repo`** (внешний стартер) → `npx degit <repo> "<target>"`, затем `cd "<target>"`.
  Если degit упал (репо приватный/не найден) → объясни просто, предложи альтернативу.
- **`scaffold_cmd`** (официальный CLI) → выполни команду, подставив `<target>`
  (напр. `npx create-next-app@latest "<target>" --ts --tailwind --app --use-npm`),
  затем выполни `post` шаги рецепта (напр. установка supabase-клиента).
- **`action: skill`** → НЕ создавай проект. Если навык доступен в твоей среде —
  запусти его, передав собранные из `ask` ответы как вход. Если навыка в среде нет —
  собери артефакт напрямую своими силами (дизайн/фронтенд), не блокируйся.
- **`action: mcp`** → НЕ создавай проект. Алгоритм:
  1) убедись, что MCP из `needs.mcp` подключён; нет → установи (`lib/mcp-install.md`);
  2) нужен ключ → спроси у резидента, сохрани;
  3) полное имя инструмента = `mcp__<server>__<mcp_tool>` (server из `needs.mcp[0]`);
  4) для async-инструментов (видео): вызвать → дождаться (`wait_for_task`/`get_task_status`)
     → при необходимости получить финал (напр. `veo3_get_1080p_video`) → сохранить файл;
  5) установка невозможна (закрытый доступ) → объясни и предложи альтернативу.

**После копирования кода (template/repo/scaffold_cmd):**
- Замени ВСЕ плейсхолдеры `{{...}}` в `src/`, `README.md`, `package.json` (поле `name`),
  `AGENTS.md`, `PROJECT-BRIEF.md` значениями из интервью и config.
- Выполни установку зависимостей (`cd "<target>" && npm install`), если есть `package.json`.

## ШАГ 6. ФИНАЛИЗАЦИЯ

### Ветка A (артефакт — презентация/видео/картинка): проекта НЕТ
- Сохрани готовый артефакт: если есть проект рядом — в него; иначе в
  `<projects_dir>/_artifacts/`. Тяжёлое медиа — по правилам базы знаний (если есть).
- НЕ клади `AGENTS.md`/`PROJECT-BRIEF.md` — для разового артефакта они не нужны.
- Если `knowledge_base` задан — сделай короткую запись об артефакте (`conventions/vault-link.md`).
- Покажи резиденту путь к файлу/ссылку.

### Ветка B (новый продукт) и доработка существующего
1. Положи в проект из `~/.comandos-kit/conventions/`:
   - `AGENTS.md` (из `AGENTS.md.tmpl`) — заполни плейсхолдеры.
   - `PROJECT-BRIEF.md` (из `PROJECT-BRIEF.md.tmpl`) — заполни по интервью.
   (для доработки — только если их ещё нет, иначе обнови существующие)
2. Если `knowledge_base` задан — заведи запись о проекте (`conventions/vault-link.md`).
3. Покажи: что создано/изменено, как запустить (точная 1 команда из README шаблона),
   что дальше.
4. Спроси: «Запустить локально / задеплоить, чтобы увидеть вживую?»

### Ветка C (знание)
- Сохрани в базу знаний (если есть) или в `templates/research`-структуру в `target`.
